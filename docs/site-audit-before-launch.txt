# Аудит сайта СМУ-1 перед запуском

Дата аудита: 2026-05-24

Ограничение аудита: по задаче не менялись дизайн, код, данные, стили, компоненты, маршруты, админка и контент. Единственное внесенное изменение - создан этот TXT-файл с отчетом.

Проверенные области проекта:
- публичные страницы в src/pages;
- общие layout/components: BaseLayout, Header, Footer, MobileMenu, Hero, Breadcrumbs, формы, карточки, галереи;
- данные и коллекции src/content, src/data;
- публичные assets в public;
- админка и визуальный редактор в src/pages/admin и tools/admin-api;
- SEO-обвязка, robots.txt, sitemap-сборка, canonical/OG в BaseLayout;
- сборка Astro.

Команды проверки:
- npm run build - успешно, собрано 113 страниц, sitemap-index.xml создан в dist.
- npm run check - команда не стартовала из-за ошибки окружения: windows sandbox: setup refresh failed with status exit code: 1.


1. Краткий вывод

Сайт находится в рабочем предрелизном состоянии по архитектуре: Astro SSG собирает публичные страницы, есть каталог, страницы товаров, услуги, выполненные объекты, контакты, политика конфиденциальности, админка/визуальный редактор, управление верхним меню, cookie banner, Яндекс Метрика и sitemap-генерация.

Главный вывод: технически сайт уже близок к публикации, но запускать его без короткого предрелизного цикла рискованно. Самые важные зоны риска сейчас не в самой сборке, а в поисковом запуске, индексации админки, финальном домене/canonical/sitemap, качестве реального контента и медиа, а также в визуальной полировке карточек и страниц.

Критично до публикации:
- привести robots.txt, sitemap и SITE_URL к финальному домену;
- закрыть админку и визуальный редактор от индексации;
- проверить canonical на всех публичных страницах;
- скрыть или дописать незавершенные страницы/пункты меню;
- заменить наиболее заметные технические placeholder-изображения и дефолтные формулировки;
- пройти ручную проверку mobile, hero video, форм, меню и ключевых маршрутов;
- добиться запуска npm run check или зафиксировать причину, почему он не выполняется в текущем окружении.

Готовность к запуску по ощущениям:
- архитектура и сборка: высокая;
- SEO-техническая готовность: средняя, есть критичные настройки перед доменом;
- визуальная готовность: средняя, сайт уже аккуратный, но местами выглядит как заготовка из-за placeholders и generic-контента;
- контентная готовность: средняя/низкая для полноценной коммерческой подачи, особенно по реальным объектам и фотографиям;
- админка: функциональная, но перегруженная и требует аккуратного регламента работы.


2. Текущее состояние проекта

Стек и сборка:
- проект собран на Astro с TypeScript;
- output static, подходит для GitHub Pages и статического хостинга;
- sitemap генерируется через @astrojs/sitemap;
- build прошел успешно, 113 страниц;
- npm run check не удалось запустить из-за ошибки sandbox setup, поэтому TypeScript/Astro check требует повторной проверки в нормальном локальном окружении.

Изображения:
- публичные изображения лежат в public/assets и public/uploads;
- много SVG-placeholder для категорий, товаров, объектов, production/trust;

- у выполненных объектов уже есть логика coverImage/image/gallery/images с обратной совместимостью.

Повторяющиеся компоненты:
- Header, MobileMenu, Footer;
- BaseLayout;
- Hero;
- Breadcrumbs;
- LeadForm / CTA;
- карточки категорий, товаров, объектов;
- галереи и карусели для объектов;
- CookieBanner.

Основные стили:
- src/styles/global.css - центральный файл стилей, где сосредоточены palette, layout, header, hero, карточки, формы, catalog/product/object/admin-related public styles;
- стили достаточно объемные, часть решений живет в одном global.css, что удобно для быстрого запуска, но усложняет долгосрочную поддержку.

Админка:
- src/pages/admin/index.astro - вход/обзор;
- src/pages/admin/visual.astro - основной визуальный редактор;
- src/pages/admin/catalog.astro, admin/technical.astro, admin/pages/navesy.astro - отдельные административные/технические зоны;
- tools/admin-api/server.mjs - локальный API для чтения/сохранения контента, upload, publish, navigation;
- загрузка файлов идет в public/uploads;
- есть импорт/экспорт каталога, массовые операции и редактирование объектов/страниц/навигации.

Риск текущей структуры:
- админка собирается как статические страницы и попадает в dist;
- без robots/noindex она может быть доступна поисковикам;
- визуальный редактор функционально мощный, но из-за большого количества возможностей повышает риск случайного изменения данных.


3. Дизайн

Общий дизайн:
- сайт выглядит современнее типичного малого строительного сайта: чистая сетка, спокойные цвета, аккуратная типографика, понятные карточки;
- сайт уже не выглядит полностью сырым, но еще не выглядит достаточно дорогим и убедительным для производственной/строительной компании;
- главная слабость - ощущение шаблонности там, где нет реальных фотографий, конкретных кейсов и фактуры производства;
- визуальный язык скорее "чистый корпоративный сайт", чем "сильная производственная компания с цехом, объектами и опытом".

Что работает хорошо:
- понятная структура: направления, каталог, услуги, объекты, контакты;
- аккуратная базовая сетка;
- заметный зеленый акцент;
- достаточно спокойные карточки и кнопки;
- нормальный базовый responsive-подход;
- выполненные объекты после доработок выглядят ближе к кейсам.

Что выглядит слабее:
- многие изображения - placeholder, из-за этого сайт сразу теряет доверие;
- часть карточек выглядит ровно, но слишком нейтрально, без коммерческого "веса";
- много белых поверхностей и тонких рамок, мало индустриальной фактуры;
- некоторые страницы услуг выглядят более текстовыми, чем продающими;
- карточки товаров и категорий местами похожи на технический каталог, если нет фото;
- декоративная система не до конца единая: часть страниц строится через content/pageBlocks, часть через более жестко прописанные секции;
- большие заголовки в hero могут выглядеть тяжело на мобильном и на страницах с длинными названиями.

Главная страница:
- сильная сторона: сразу есть hero, направления каталога/услуг, переход к объектам, CTA;
- риск: если hero video не загрузился или используется слабый poster, первый экран теряет эффект;
- блоки ниже читаются, но доверие сильно зависит от качества фото и конкретики;
- нужно меньше generic-обещаний и больше фактов: цех, сроки, типовые заказчики, реальные объекты, география, "что производим сами".

Каталог уличной мебели:
- структура хорошая: раздел -> категории -> товары;
- визуально готовность зависит от фото;
- если используются placeholder, каталог выглядит как заготовка, а не как ассортимент;
- карточки нужно держать едиными по высоте и плотности, иначе каталог будет "сыпаться" при разном контенте.

Каталог ограждений:
- коммерчески важный раздел, потому что ближе к B2B/ЖКХ/благоустройству;
- нужна более жесткая подача: тип объекта, монтаж, материалы, варианты исполнения, сроки;
- визуально ему не хватает фото готовых ограждений/контейнерных площадок в реальной среде.

Карточка товара:
- структура сильная: фото/галерея, характеристики, преимущества, CTA, похожие товары;
- риск - дефолтные фразы и характеристики могут выглядеть одинаково между товарами;
- если нет цены, "по запросу" нормально для B2B, но нужна явная причина: размеры, металл, покрытие, монтаж, партия.

Выполненные объекты:
- листинг стал лучше: карточки с фото/каруселью, названием, городом/годом, типом, описанием, кнопкой;
- страница объекта логически правильная: одна галерея и смысловые блоки;
- главный риск - мало реальных кейсов и мало фактуры результата;
- для доверия нужны фото до/после, детали, общий вид, монтаж, подписи.

Навесы:
- направление важно и коммерчески сильное;
- нужно проверить, нет ли дубля логики /navesy/ и /navesy-i-kozyrki/ с разным смыслом;
- визуально нужны большие реальные фото навесов, иначе раздел воспринимается как абстрактная услуга.

Металлоконструкции:
- для B2B это ключевой раздел;
- текущая подача должна сильнее показывать производственные возможности: сварка, размеры, партии, чертежи, монтаж, работа с юрлицами;
- дизайн должен быть более "инженерным": таблицы возможностей, фото цеха, типовые задачи.

Контакты:
- критичная страница для запуска;
- хорошо, что есть телефон, email, Telegram, адрес/реквизиты;
- нужно проверить кликабельность tel/mail/telegram и актуальность всех реквизитов;
- желательно добавить "как быстрее получить расчет": фото/эскиз, размеры, город, срок, количество.

Шапка:
- шапка уже стала управляемой через данные;
- на desktop меню может быть перегружено при большом числе пунктов;
- нужно держать в меню только самые важные разделы, остальное уводить в footer или вторичные ссылки;
- телефон и Telegram должны оставаться видимыми, потому что это главный путь заявки.

Футер:
- должен быть не просто дублированием меню, а служебной картой сайта: направления, документы, контакты, реквизиты;
- важно проверить, что в футере нет устаревших или битых ссылок.

Формы и CTA:
- CTA есть, но их нужно привязать к контексту страниц;
- на карточке товара CTA должен звучать как "Рассчитать изделие" или "Запросить расчет", а не абстрактно;
- на услугах - "Обсудить объект";
- на объектах - "Хочу похожее решение";
- сейчас часть CTA может восприниматься слишком общей.


4. Цветовая палитра

Текущая палитра:
- основа светлая: белый/молочный фон, темный графитовый текст, серые вторичные тексты, зеленый акцент;
- для СМУ-1 это рабочая база: аккуратно, спокойно, не кричаще;
- зеленый акцент хорошо связан с благоустройством и городской средой;
- для производственной компании палитре не хватает "материальности": стали, графита, теплого металла, более уверенных темных поверхностей;
- на страницах с большим количеством placeholder светлая палитра усиливает ощущение пустоты;
- контраст в основных зонах нормальный, но нужно отдельно проверить мелкие подписи, бейджи, placeholder-тексты и вторичные кнопки.

Где палитра может конфликтовать:
- зеленый акцент подходит благоустройству, но не всегда поддерживает металлоконструкции/сварочный цех;
- светлые карточки на светлом фоне иногда слишком мягкие;
- если фото будут цветными и разными по качеству, текущая палитра может не удерживать цельность без более строгих рамок/фонов.

Направление А: оставить текущую основу и аккуратно усилить

Цвета:
- основной фон: #FFFFFF;
- вторичный фон: #F5F7F8;
- индустриальный светлый фон: #EEF3F1;
- основной текст: #11161A;
- вторичный текст: #5C6670;
- линия/бордер: #D8E0E4;
- акцент: #18B663;
- темный акцент для hover: #10914E.

Где применять:
- оставить текущие светлые страницы;
- усилить рамки карточек и фон отдельных секций;
- зеленый использовать только для CTA, ссылок, активных состояний, важных бейджей;
- добавить графитовые CTA/hero-фрагменты на производственных страницах.

Плюсы:
- низкая сложность;
- сохраняет текущий сайт;
- быстро улучшает собранность;
- не требует полной переработки дизайна.

Минусы:
- не даст резкого премиального эффекта;
- при слабых фото сайт все равно будет выглядеть как аккуратная заготовка.

Подходит ли СМУ-1:
- да, это лучший вариант для предзапусковой полировки без риска.

Цвета:

- стальной: #2E3A44;
- теплый светлый фон: #F4F1EA;

- акцент 1: #18B663;
- акцент 2, дозировано: #B88746 или #C49A5A.

Где применять:
- hero на главной и B2B-направлениях;

- CTA-плашки;
- карточки ключевых преимуществ;
- футер.

Плюсы:
- сайт будет выглядеть дороже и увереннее;
- лучше поддерживает сварочный цех, металл, промышленное производство;
- реальные фото объектов на темном/графитовом фоне будут смотреться сильнее.

Минусы:
- выше сложность;
- нужен аккуратный контроль контраста;
- при плохих фото темная палитра может выглядеть тяжело;
- больше риск сломать текущую визуальную систему.

Подходит ли СМУ-1:
- подходит хорошо, но лучше делать после запуска или отдельным этапом редизайна.

- страницы с большими фотографиями;
- карточки, где важны чистота, масштаб и аккуратность.

Плюсы:
- выглядит современно и спокойно;
- хорошо подходит городской среде, архитектуре, благоустройству;
- не требует темного тяжелого дизайна.

Минусы:

- требует сильных фотографий и строгой сетки;

Подходит ли СМУ-1:
- подходит для направления благоустройства и уличной мебели; для сварочного цеха лучше добавить графитовые индустриальные акценты.


5. Типографика

Текущее состояние:
- связка заголовочного шрифта и текстового шрифта читается современно;
- текстовые блоки в целом читаемые;
- заголовки местами слишком крупные для коммерческих внутренних страниц;
- длинные H1 могут давать тяжелые переносы на mobile;
- между страницами есть риск разного ощущения масштаба, потому что часть страниц собрана через универсальные компоненты, а часть через специальные секции;
- карточки могут страдать от разных длин названий, особенно в каталоге и объектах.

H1:
- на главной крупный H1 уместен, если hero визуально сильный;
- на внутренних страницах H1 стоит держать компактнее;
- длинные названия товаров, категорий и услуг должны иметь ограничение по ширине и адекватные mobile-размеры;
- важно избегать ситуации, когда первый экран почти полностью занят заголовком.

H2:
- H2 должны быть меньше hero-заголовков и единообразны между страницами;
- часть H2 стоит делать более коммерческими: не "Преимущества", а "Что важно для объекта", "Как рассчитывается стоимость", "Что входит в изготовление".

Карточки:
- заголовки карточек должны иметь стабильную высоту или line-clamp;
- описание карточек нужно ограничивать 2-3 строками;
- бейджи не должны конкурировать с заголовком;
- на mobile карточка должна читаться сверху вниз: фото, тип/категория, название, описание, действие.

Описания:
- слишком длинные абзацы в каталогах и услугах нужно дробить на короткие смысловые блоки;
- на B2B-страницах лучше работают списки "что делаем / для кого / что влияет на цену".

Кнопки:
- текст кнопок должен быть конкретным;
- "Подробнее" допустимо для карточки, но в коммерческих CTA лучше "Запросить расчет", "Обсудить объект", "Рассчитать изделие";
- размеры кнопок на mobile должны быть не меньше удобной зоны нажатия.

Бейджи и подписи:
- бейджи должны поддерживать сканирование, а не быть декоративными;
- подписи к фото нужны только если дают смысл: объект, город, этап, материал, год;
- технические подписи и имена файлов нельзя выводить публично.

Mobile typography:
- проверить H1 на 360-390 px ширине;
- проверить длинные слова: "металлоконструкции", "контейнерных", "благоустройство";
- проверить, не становятся ли вторичные подписи слишком мелкими;
- проверить, не переносится ли телефон/кнопки в шапке некрасиво.


6. UX и структура

Понятность предложения:
- по структуре сайта понятно, что компания занимается уличной мебелью, ограждениями, навесами, металлоконструкциями, благоустройством и строительными работами;
- коммерческий фокус на загрузку сварочного цеха стоит усилить: сейчас он читается не везде;
- на главной и в B2B-разделах нужно явнее сказать, что СМУ-1 производит/изготавливает, а не просто продает каталог.

Логика движения пользователя:
- текущая цепочка работает: главная -> каталог/услуги/объекты -> карточка/раздел -> заявка;
- не везде хватает промежуточных CTA после блоков доверия и характеристик;
- пользователь из карточки товара должен быстро понять, что нужно отправить для расчета;
- пользователь из объекта должен видеть переход "хочу похожее решение".

Куда нажимать:
- верхнее меню понятное, но перегруженное при большом числе пунктов;
- карточки товаров/объектов должны всегда иметь явное действие;
- в каталоге хорошо бы усилить визуальное различие между карточкой категории и карточкой товара.

Заказ расчета:
- формы и CTA присутствуют;
- нужно проверить, что форма передает достаточно контекста: страница, товар/услуга, URL;
- важно добавить подсказку рядом с формой: "для расчета приложите размеры, фото места, количество, город";
- если используется внешний form backend, перед запуском нужно проверить доставку заявок.

Категории:
- структура категорий понятная, но местами широкий каталог требует фильтров или хотя бы быстрых якорей;
- на старте можно обойтись без фильтров, если карточек не слишком много и сетка аккуратная;
- при росте ассортимента нужен поиск/фильтр по типу изделия, материалу, назначению.

Выполненные объекты:
- раздел должен работать как доказательство компетенции, не просто как галерея;
- карточки должны показывать результат, город/год, тип работ;
- внутри объекта нужны: задача, что сделали, материалы, особенности, фото результата;
- полезно связать объект с товарными категориями: "использованы лавочки", "ограждение", "навес".

Доверие:
- сейчас доверие зависит от контента;
- нужны реальные фото цеха, объектов, команды/процесса, монтажа;
- нужны факты: работаем с юрлицами, собственное производство, гарантия, порошковая окраска, монтаж, доставка по региону/России;
- реквизиты и контакты есть, их нужно сделать легко проверяемыми.

Что добавить:
- блок "Что нужно для расчета" на товарных и услуговых страницах;
- блок "Производим сами" с фото цеха;
- связанные объекты на страницах услуг и товаров;
- мини-FAQ по цене/срокам/доставке/монтажу;
- CTA после галереи объекта;
- страницы/секции под локальные запросы: Курган, Курганская область, поставка по России.

Что убрать или сократить:
- повторяющиеся generic-блоки, которые не добавляют фактов;
- технические placeholders в первом экране или карточках;
- пункты меню, которые не нужны в верхней навигации;
- слишком длинные вводные тексты до карточек/CTA.

Где нужна навигация назад:
- карточка товара -> категория;
- объект -> список объектов;
- категория ограждений/мебели -> раздел;
- админка -> список редактируемого типа данных.


7. Каталог

Структура:
- каталог построен через product-sections, product-categories, products;
- есть разделы уличной мебели и ограждений, а также отдельные направления навесов, металлоконструкций, топиариев;
- URL в целом человекопонятные и транслитерированные;
- у товаров есть отдельные страницы, что хорошо для SEO и коммерческой выдачи.

Карточки типов изделий:
- карточки категорий дают быстрый вход в ассортимент;
- их качество сильно зависит от иллюстраций;
- placeholder на карточке категории делает раздел менее убедительным;
- названия должны быть короткими и не ломать сетку.

Карточки товаров:
- наличие отдельных товаров - сильная сторона;
- карточки должны показывать фото, назначение, короткое описание, цену/по запросу, CTA;
- при одинаковых default-текстах карточки могут выглядеть размноженными;
- важно ограничить описания и держать одинаковую высоту.

Изображения:
- главный риск каталога - недостаток реальных фото;
- для запуска нужно хотя бы по одному сильному изображению на ключевую категорию и топовые товары;
- для товара лучше иметь 3-5 фото: общий вид, деталь, материал/покрытие, изделие в среде, схема/габариты;
- технические SVG-placeholder допустимы только временно, но не на главных коммерческих позициях.

Описание и контент:
- нужно убрать или сократить дефолтные фразы там, где они повторяются массово;
- для каждого важного товара нужно добавить: назначение, размеры/вариативность, материал, покрытие, способ крепления, возможность изготовления под размер;
- "по запросу" нужно объяснить: цена зависит от размеров, металла, покрытия, партии, доставки и монтажа.

Материалы и характеристики:
- характеристики важны для B2B и закупок;
- стоит привести характеристики к единому набору полей;
- для уличной мебели: металл, дерево/ДПК, покрытие, крепление, габариты, вес при наличии;
- для ограждений: высота, секция, профиль, покрытие, монтаж, варианты ворот/калиток;
- для навесов: каркас, покрытие, кровля, крепление, снеговая нагрузка при наличии.

Индивидуальное изготовление:
- это сильное коммерческое преимущество СМУ-1;
- сейчас его нужно выводить не только на отдельной странице, но и в карточках товаров;
- CTA должен вести к расчету под объект.

Похожие товары:
- полезный блок, если он не выводит случайные или слишком похожие карточки;
- лучше показывать товары из той же категории или смежные изделия;
- для SEO и UX блок полезен, но не должен перегружать страницу.

Готовность каталога к запуску:
- технически каталог готов;
- визуально и коммерчески готовность зависит от замены placeholder и чистки generic-текстов;
- для запуска можно оставить часть вторичных товаров с минимальным контентом, но ключевые направления должны быть доведены.

Что мешает восприятию:
- placeholder-изображения;
- одинаковые описания;
- не всегда очевидная разница между товарами;
- отсутствие реальных примеров использования;
- недостаток доверительных фактов рядом с CTA.

Что критично до запуска:
- проверить ключевые страницы: уличная мебель, лавочки, урны, ограждения, контейнерные площадки, навесы, металлоконструкции;
- заменить top-level placeholders;
- проверить, что все видимые товары действительно нужны в запуске;
- скрыть или дооформить слабые позиции.

Что можно после запуска:
- фильтры;
- расширенные сравнения;
- PDF-каталог;
- больше характеристик;
- схемы/чертежи;
- интеграция с CRM/заявками.


8. Выполненные объекты

Текущее состояние:
- список объектов существует на /vypolnennye-obekty/;
- сборка показывает 2 отдельных объекта;
- карточки объектов уже имеют более коммерческую структуру: фото/карусель, название, город/год, тип работ, описание, кнопка;
- страница отдельного объекта использует одну большую галерею, а не два независимых фото-блока;
- галерея поддерживает несколько фото и placeholder;
- вертикальные фото должны отображаться через blurred background + foreground contain.

Сильные стороны:
- раздел уже готов как техническая основа для кейсов;
- есть обратная совместимость с image/coverImage/gallery/images;
- карточки кликабельны и понятнее, чем изначально;
- страница объекта подходит для будущего наполнения реальными фото.

Слабые стороны:
- всего 2 объекта мало для доверительного контура;
- если фото остаются placeholder, раздел не выполняет свою главную функцию;
- текстовая структура пока скорее базовая, чем полноценный кейс;
- нет явной связи объектов с товарами/категориями/услугами;
- мало полей, которые помогают заказчику понять масштаб работ.

Как лучше оформить кейсы:
- первый экран: название, город/год, тип работ, крупное фото результата;
- галерея: общий вид, детали, монтаж, до/после, изделие в среде;
- ниже: задача, что сделали, материалы, сроки/объем, особенности, результат;
- в конце: CTA "Хотите похожий объект? Отправьте размеры или фото площадки";
- добавить связанные категории/товары: лавочки, ограждения, навесы, контейнерная площадка.

Нужные поля:
- title;
- slug;
- city;
- year;
- workType/category;
- shortDescription;
- task/problem;
- worksDone;
- materials;
- features;
- coverImage;
- gallery/images;
- captions optional;
- relatedCategories/products/services;
- isPublished или скрытие из публичной части, если админка это поддержит.

Лишние поля:
- отдельные технические подписи, если они не выводятся публично;
- дублирующие image/coverImage без нормализации;
- отдельные плитки под каждый маленький факт, если это перегружает страницу.

Какие фото нужны:
- горизонтальное обложечное фото для карточки и OG;
- 3-8 фото на объект;
- вертикальные фото допустимы, но лучше не делать их единственным материалом;
- нужны фото деталей металла/покрытия/крепления;
- нужны фото объекта в окружении, не только крупный фрагмент изделия.

Как показывать вертикальные фото:
- текущий прием blurred background + contain правильный;
- важно затемнять/успокаивать фон, чтобы он не спорил с основным фото;
- media-блок должен иметь фиксированное соотношение сторон, чтобы не было layout shift;
- placeholder не должен использовать blurred background.

Как усилить доверие через кейсы:
- добавить реальные цифры: город, год, тип заказчика, объем, количество изделий;
- добавить "что было важно" и "какое решение выбрали";
- показать монтаж и итоговый вид;
- связать кейсы с конкретными направлениями каталога;
- вывести 3-4 лучших объекта на главной.


9. Админка

Текущее состояние:
- есть визуальный редактор с редактированием контента, объектов, меню, медиа;
- есть загрузка изображений/видео в public/uploads;
- есть publish-механизм через Git;
- есть импорт/экспорт каталога;
- есть управление пунктами шапки через navigation.json;
- админка достаточно функциональна для статического сайта.

Удобство наполнения:
- плюс: многие данные можно менять без ручного редактирования файлов;
- плюс: можно менять меню, фото, объекты, контент страниц;
- минус: визуальный редактор перегружен, новичку легко потеряться;
- минус: рядом находятся безопасные контентные настройки и более рискованные технические операции;
- минус: если нет строгой валидации, можно сохранить битую ссылку, пустой title или неправильный путь картинки.

Навигация:
- редактирование меню в админке полезно и решает проблему перегруженной шапки;
- нужно добавить/проверить валидацию URL: внутренние пути должны начинаться с /, внешние - с https://;
- порядок и скрытие пунктов работают как нужная базовая CMS-логика;
- скрытие пункта меню не должно удалять страницу, это правильно.

Импорт/экспорт каталога:
- полезно для массовой работы;
- высокий риск сломать данные при неправильном JSON;
- до запуска нужен backup и понятная инструкция;
- желательно ограничить доступ к этим операциям.

Выполненные объекты:
- админка поддерживает работу с галереей;
- это нужно сохранить, но добавить регламент: размеры фото, имена файлов, обложка, подписи;
- желательно иметь preview объекта перед публикацией.

Фото:
- загрузка в uploads есть;
- нет очевидной автоматической оптимизации веса/формата;
- до запуска нужно вручную сжимать фото;
- после запуска желательно добавить image optimization pipeline.

Риски:
- админка собирается в публичный dist;
- если она индексируется, это плохо для SEO и безопасности восприятия;
- локальный API нельзя считать полноценной защищенной CMS;
- dev credentials должны быть заменены/выключены в production-сценарии;
- slug-редактирование может ломать ссылки и SEO.

Критично до запуска:
- закрыть /admin/ от индексации через robots и/или meta noindex;
- проверить, что admin API не деплоится как открытый backend, если сайт статический;
- проверить пароли/ENV для локальной админки;
- сделать backup src/content и src/data перед массовым наполнением;
- проверить publish workflow.

Желательно после запуска:
- разделить редактор на простые разделы: страницы, каталог, объекты, меню, медиа, техоперации;
- добавить валидацию ссылок и обязательных полей;
- добавить preview перед publish;
- добавить массовую оптимизацию изображений;
- добавить статусы "черновик/опубликовано".

Можно потом:
- роли пользователей;
- история изменений;
- drag-and-drop галерей везде;
- медиа-библиотека;
- автоматическая проверка битых ссылок.


10. Мобильная версия

Общее состояние:
- проект явно учитывает mobile: есть мобильное меню, responsive hero, сетки, карточки;
- сборка не выявляет технических ошибок;
- финальный mobile-QA на реальном телефоне обязателен, особенно из-за hero video, меню, каруселей и длинных русских заголовков.

Главная:
- главный риск - видео в hero;
- fallback/poster должен всегда быть виден, если autoplay не сработал;
- H1 не должен занимать слишком много высоты;
- CTA в первом экране должен быть виден без лишней прокрутки.

Каталог:
- карточки должны идти в один столбец или аккуратную 2-колоночную сетку только там, где хватает ширины;
- длинные названия товаров нужно проверить на 360 px;
- описания не должны создавать слишком высокие карточки;
- кнопки должны быть легко нажимаемыми.

Карточки товара:
- галерея/фото не должны создавать горизонтальный overflow;
- таблицы характеристик должны нормально переноситься;
- CTA и форма должны быть близко к ключевой информации;
- похожие товары не должны чрезмерно растягивать страницу.

Выполненные объекты:
- карточная карусель должна иметь удобные стрелки/индикаторы;
- media-блок должен быть стабильным;
- вертикальные фото должны отображаться без белых пустот;
- кнопка "Смотреть объект" должна оставаться видимой.

Меню:
- мобильное меню берет управляемые пункты из navigation.json;
- нужно проверить сценарий, когда в меню много пунктов;
- телефон и Telegram должны быть доступны;
- body scroll lock/close behavior нужно проверить вручную.

Hero video:
- реализованы autoplay/muted/loop/playsinline/preload/poster и повторные попытки play;
- полностью гарантировать autoplay на iPhone нельзя;

13.1. Индексация, поисковики, Яндекс Вебмастер и Google Search Console

robots.txt:
- файл есть в public/robots.txt;
- текущая логика Allow: / открывает сайт целиком;
- риск: /admin/ не закрыт от индексации;
- риск: sitemap URL в robots.txt выглядит временным и не привязан к финальному домену;
- нужно добавить Disallow для /admin/ и, при необходимости, других технических зон;
- нужно указать корректный Sitemap на финальном домене;
- конфликтов Disallow/Allow сейчас нет, но правил слишком мало для production.

sitemap.xml / sitemap-index.xml:
- sitemap генерируется при npm run build;
- build сообщил: sitemap-index.xml created at dist;
- в sitemap должны попадать публичные страницы, категории, товары, услуги, объекты, контакты, политика;
- нужно проверить, что туда не попадают админка и служебные страницы;
- если Astro sitemap видит все сгенерированные страницы, админские HTML могут попасть в sitemap, это критично проверить;
- absolute URL зависят от SITE_URL;
- после подключения основного домена sitemap не должен ссылаться на koleaw.github.io, example.com или временный GitHub Pages URL.

Canonical URL:
- canonical строится в BaseLayout через Astro.site и Astro.url.pathname;
- это правильный подход только при правильном SITE_URL;
- если SITE_URL останется example.com или GitHub Pages, canonical будет неправильным;
- после подключения основного домена нужна единая логика canonical для всех публичных страниц;
- нужно проверить, что query/hash не создают дублей и что trailing slash consistent.

Meta robots / noindex:
- публичные страницы не должны иметь случайный noindex;
- админка, визуальный редактор, технические страницы должны иметь noindex и/или быть закрыты robots;
- сейчас критичный риск - статические admin pages в dist;
- страницы редактора не должны индексироваться и не должны быть в sitemap.

Title/description по типам страниц:
- главная: есть, но нужно проверить коммерческую силу и локальность;
- разделы каталога: есть, но требуют уникальности и пользы;
- категории товаров: должны различаться по назначению и ключам;
- карточки товаров: нужна уникальность, иначе будет массовая шаблонность;
- выполненные объекты: нужны конкретные title/description с городом/типом работ;
- контакты: должны содержать СМУ-1, Курган, телефон, производство/металлоконструкции;
- о компании: должна усиливать доверие, не быть общей биографией.

H1-H2:
- перед запуском нужен crawl или скрипт проверки H1 count;
- один понятный H1 на публичную страницу;
- H2 должны быть структурными, не декоративными;
- карточки/логотипы/скрытые mobile-меню не должны создавать лишние H1.

URL и slug:
- человекопонятность в целом хорошая;
- нужно проверить опечатки, транслитерацию и дубли;
- скрытые/неопубликованные элементы не должны быть доступны, если это предусмотрено;
- если элемент скрыт только из меню, но страница публичная, это нормально, но нужно понимать SEO-последствия.

404 и редиректы:

- реквизиты: сверить ИНН, КПП, ОГРН, юридический адрес;
- адрес/карта: проверить соответствие фактической зоне работы.

Формы:
- нужно проверить, куда уходят заявки;
- должна быть защита от спама;
- должен быть понятный post-submit текст;
- согласие/политика в форме должны соответствовать фактической юридической логике.

16. Приоритеты

Критично до запуска

Пункт: robots.txt и закрытие админки.
Почему важно: admin pages собираются в dist и не должны попадать в индекс.
Где исправлять: public/robots.txt, src/pages/admin/*.astro или BaseLayout/noindex logic, sitemap config.
Риск если не исправить: поисковики увидят служебные страницы, ухудшится качество индекса и доверие.

Пункт: SITE_URL/canonical/sitemap под финальный домен.
Почему важно: неправильный canonical может закрепить в поиске временный домен.
Где исправлять: astro.config.mjs, env SITE_URL, public/robots.txt, деплой.
Риск если не исправить: дубли, плохая индексация, проблемы при переезде на домен.

Пункт: реальные изображения на ключевых страницах.
Почему важно: placeholder снижает доверие и делает сайт похожим на незавершенный.
Где исправлять: public/assets, public/uploads, src/content products/categories/projects/static-pages.
Риск если не исправить: ниже конверсия, слабый визуальный образ компании.

Пункт: проверка форм и контактов.
Почему важно: сайт должен принимать заявки сразу после запуска.
Где исправлять: компоненты форм, внешний form backend, site-settings.
Риск если не исправить: потеря заявок.

Пункт: npm run check в рабочем окружении.
Почему важно: build прошел, но check не подтвердил типовую корректность.
Где исправлять: локальное окружение/CI, TypeScript/Astro ошибки при наличии.
Риск если не исправить: скрытые ошибки типов и контентных схем.

Пункт: mobile QA.
Почему важно: значительная часть пользователей откроет сайт с телефона.
Где исправлять: global.css, Header/MobileMenu, Hero, cards, galleries, forms.
Риск если не исправить: плохой первый опыт, потери заявок.

Важно, но можно после запуска

Пункт: микроразметка Organization/BreadcrumbList.
Почему важно: улучшает понимание сайта поисковиками.
Когда делать: сразу после базовой индексационной настройки или в коротком post-launch.

Пункт: улучшение текстов категорий и товаров.
Почему важно: повышает SEO и конверсию.
Когда делать: итерационно, начиная с ключевых направлений.

Пункт: больше выполненных объектов.
Почему важно: кейсы - главный доверительный аргумент.
Когда делать: по мере подготовки фото и описаний.

Пункт: оптимизация изображений.
Почему важно: ускоряет mobile и LCP.
Когда делать: до массовой загрузки реальных фото, но можно начать вручную.

Пункт: связка товаров и объектов.
Почему важно: пользователь видит изделия в реальных проектах.
Когда делать: после наполнения 5-10 кейсов.

Улучшения на будущее

Пункт: фильтры и поиск в каталоге.
Эффект: быстрее находить изделия.
Сложность: средняя.

Пункт: медиа-библиотека в админке.
Эффект: удобнее управлять фото и избегать дублей.
Сложность: средняя/высокая.

Пункт: автоматическая оптимизация uploads.
Эффект: стабильная производительность.
Сложность: средняя.

Пункт: IndexNow.
Эффект: быстрее уведомлять поисковики о новых страницах.
Сложность: низкая/средняя.

Пункт: расширенная аналитика целей.
Эффект: понятнее, какие страницы дают заявки.
Сложность: низкая.


17. Концепции улучшения дизайна

Концепция 1: аккуратная полировка текущего дизайна

Что меняем:
- усиливаем контраст карточек;
- приводим отступы между страницами к единой шкале;
- уменьшаем чрезмерные заголовки на внутренних страницах;
- улучшаем media-блоки карточек;
- добавляем реальные фото на ключевые страницы;
- аккуратно усиливаем CTA.

Что оставляем:
- текущую светлую основу;
- зеленый акцент;
- структуру страниц;
- текущую шапку/футер;
- Astro/content architecture.

Палитра:
- белый/светло-серый фон;
- графитовый текст;
- зеленый CTA;
- чуть более заметные линии и тени.

Карточки:
- стабильные media-блоки;
- меньше пустоты;
- четкая иерархия: фото, тип, название, описание, действие;
- hover/focus без агрессивных эффектов.

Шапка:
- оставить управляемое меню;
- скрыть второстепенные пункты;
- держать телефон и Telegram заметными.

Hero:
- оставить текущий формат;
- улучшить poster/video;
- сделать первый экран более предметным.

Кнопки:
- меньше generic;
- больше конкретики под контекст.

Ожидаемый эффект:
- сайт будет выглядеть собраннее и надежнее без полного редизайна.

Сложность:
- низкая/средняя.

Файлы:
- src/styles/global.css;
- src/components/Header.astro;
- src/components/Hero.astro;
- карточки/галереи;
- src/content/static-pages, products, projects.

Риски:
- минимальные, если работать точечно.

Подходит ли СМУ-1:
- да, лучший вариант до запуска.

Делать до запуска или после:
- часть критичных правок до запуска, остальное после.

Концепция 2: более премиальный индустриальный стиль

Что меняем:
- добавляем графитовые/стальные секции;
- усиливаем роль фото производства и металла;
- делаем CTA более контрастными;
- строим страницы вокруг "производим, варим, монтируем, отвечаем";
- объекты и производство подаем крупнее.

Палитра:
- графит, сталь, светлый теплый фон;
- зеленый как функциональный CTA;
- можно добавить теплый металл/латунь дозировано.

Больше темных/графитовых элементов:
- hero;
- производство;
- металлоконструкции;
- футер;
- крупные CTA.

Как подать металл, производство, объекты:
- реальные фото цеха;
- детали сварки/покрытия;
- крупные кейсы;
- технические характеристики в строгих таблицах;
- факты вместо общих обещаний.

Плюсы:
- сильнее B2B-позиционирование;
- дороже и увереннее;
- лучше подходит сварочному цеху.

Минусы:
- больше риск;
- нужна хорошая фотобаза;
- потребуется больше дизайн-работы.

Сложность:
- средняя/высокая.

Файлы:
- src/styles/global.css;
- Header/Footer/Hero;
- главная;
- страницы услуг;
- карточки товаров;
- объекты;
- контент и изображения.

Риски:
- без качественных фото темный стиль будет выглядеть тяжелым;
- можно ухудшить читабельность.

Подходит ли СМУ-1:
- да, особенно для B2B и металлоконструкций.

Делать до запуска или после:
- после запуска как отдельный этап, если нет времени и фото сейчас.

Концепция 3: современный архитектурный светлый стиль

Что меняем:
- больше воздуха;
- строже сетка;
- крупнее реальные фото;
- меньше декоративного шума;
- карточки становятся похожими на архитектурное портфолио;
- объекты выводятся как главная доверительная витрина.

Палитра:
- теплый светло-серый фон;
- белые поверхности;
- глубокий зеленый;
- графитовый текст;
- нейтральные архитектурные линии.

Больше воздуха:
- увеличить межсекционные расстояния;
- убрать лишние мелкие декоративные элементы;
- дать фото работать как главный визуальный аргумент.

Строгая сетка:
- единые max-width;
- единые пропорции карточек;
- аккуратные галереи;
- строгая вертикальная ритмика.

Крупные фото:
- обязательны реальные объекты и изделия;
- placeholder почти недопустимы.

Плюсы:
- современный и спокойный вид;
- хорошо для благоустройства и уличной мебели;
- меньше риска перегрузить пользователя.

Минусы:
- без фото выглядит пусто;
- может быть слишком мягким для промышленной части;
- требует дисциплины контента.

Сложность:
- средняя.

Файлы:
- src/styles/global.css;
- Hero;
- Product cards;
- Object cards/gallery;
- static-pages content;
- изображения.

Риски:
- нужно много качественных изображений;
- часть производственного характера может потеряться.

Подходит ли СМУ-1:
- подходит для городской мебели и объектов; для металлоконструкций нужна индустриальная добавка.

Делать до запуска или после:
- после запуска или как мягкую полировку текущего дизайна.


18. Рекомендуемый план работ

Этап 0: предзапусковая техническая фиксация, 1 день
- определить финальный домен;
- настроить SITE_URL и BASE_PATH;
- исправить robots.txt;
- закрыть /admin/ от индексации;
- проверить sitemap, что нет admin pages;
- проверить canonical;
- повторить npm run build и npm run check;
- проверить 404;
- проверить формы;
- проверить Метрику;
- проверить основные ссылки.

Этап 1: контент и доверие, 1-3 дня
- заменить ключевые placeholder на реальные фото;
- подготовить нормальный poster для hero video;
- добавить фото производства;
- дооформить 3-5 выполненных объектов;
- проверить тексты главной, каталога, объектов, контактов;
- скрыть из меню незавершенные разделы;
- убрать явные generic-фразы на ключевых страницах.

Этап 2: mobile и производительность, 1 день
- проверить сайт на iPhone/Safari и Android/Chrome;
- проверить hero video, poster, меню, формы, галереи;
- сжать видео;
- сжать реальные фото;
- проверить Lighthouse mobile;
- проверить отсутствие горизонтального overflow.

Этап 3: поисковый запуск, день публикации
- опубликовать на финальном домене;
- проверить HTTPS и canonical;
- открыть robots.txt и sitemap-index.xml;
- добавить сайт в Яндекс Вебмастер;
- добавить sitemap;
- добавить сайт в Google Search Console;
- проверить Метрику;
- запросить переобход ключевых страниц.

Этап 4: первые 2 недели после запуска
- смотреть заявки и поведение в Метрике;
- отслеживать ошибки обхода;
- добавлять реальные кейсы;
- улучшать title/description ключевых страниц;
- улучшать фото товаров;
- добавить Organization/BreadcrumbList structured data;
- начать работу с Яндекс Бизнес/2ГИС/справочниками.

Этап 5: развитие после запуска
- улучшить дизайн по концепции 1;
- затем выбрать, нужен ли индустриальный или архитектурный редизайн;
- добавить фильтры каталога;

- автоматизировать оптимизацию изображений;
- улучшить админку: preview, validation, media library, draft/published.

Первым рекомендую делать не визуальный редизайн, а поисково-техническую подготовку:
1. SITE_URL/canonical/sitemap/robots.
2. Noindex/Disallow для админки.
3. Проверка форм и Метрики.
4. Реальные фото на первом экране, в каталоге и объектах.
5. Mobile QA на реальном телефоне.

После этого сайт можно выпускать в свет с контролируемым списком post-launch улучшений.
