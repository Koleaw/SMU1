# План реализации сайта СМУ-1 (для этапа сборки)

## 1) Что это за проект
- Сайт СМУ-1: B2B-first, со вторым приоритетом B2C.
- Коммерческий фокус: загрузка сварочного цеха.
- Три отдельные логики продаж: каталог изделий, проектные услуги, доверительный контур.
- Главный source of truth: `docs/MASTER_SPEC.md`; PDF — только для уточнений.

## 2) Стек (основной вариант)
- Astro (SSG) + TypeScript + JSON/YAML для данных + SCSS.
- Деплой: GitHub Actions -> GitHub Pages.
- Формы: внешний form backend (Formspree/Web3Forms/аналог), без серверной части в репозитории.

Почему:
- Надежно и просто для статической публикации.
- Удобно собирать страницы из шаблонов и сущностей.
- Легко поддерживать SEO и мобильную версию без тяжелого JS.

## 6) Контент и данные
- Тексты: `src/data/*.json` (при необходимости часть в markdown).
- Фото: `public/assets/images/...` (products, objects, production, placeholders).
- Данные категорий/объектов/FAQ/форм: отдельные JSON по полям из MASTER_SPEC.

## 7) Формы
- 3 типа: shortLead, calculationForm, customOrderForm.
- Единый понятный пост-submit текст: «Свяжемся, уточним задачу и подготовим расчет.»
- Антиспам: honeypot + ограничение на стороне внешнего сервиса.

## 8) Деплой
- GitHub Actions: build Astro -> публикация в GitHub Pages.
- Корректный `base`/`site` для репозитория.
- `sitemap.xml`, `robots.txt`, OG и базовый SEO-набор на всех страницах.

## 9) Что остается заглушками до мая
- Часть карточек объектов (если нет фактуры/фото).
- Часть галерей товарных категорий (placeholders).
- Цены в форматах, где пока только «по запросу».
- Временный favicon до утверждения упрощенной иконки.

## 10) Риски до старта
1. В спецификации указан путь `docs/source/`, фактически PDF лежат в `docs/`.
2. Папка `public/assets/brand/` отсутствует — нужно создать и перенести туда основной SVG.
3. Нужен ранний выбор провайдера форм и согласование юридического текста.
4. Нельзя расширять архитектуру страниц до заполнения утвержденного каркаса.
